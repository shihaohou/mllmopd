# T1 OPD Baseline Implementation Plan

**Captured at**: 2026-05-19. **Status**: ready to execute, awaits user trigger.

This is the concrete implementation plan for T1, the first OPD training run.
Goal: validate whether vanilla OPD actually transfers the teacher's
vision-conditioned capability, or whether it's mostly language-prior
distillation. The test is a **FullTeacher vs BlankTeacher OPD negative
control** — both arms share student model, prompt pool, optimizer schedule,
and token budget; only the image the teacher conditions on while scoring
the student's rollout differs.

See `docs/finding-2026-05-19-vision-conditioned.md` for the audit findings
that motivate this design.

---

## 0. Decision-tree this experiment will answer

| Outcome | Interpretation | Next step |
|---|---|---|
| T1-2 (Full) ≫ T1-3 (Blank) on math/chart benchmarks AND on opd_target subset | Visual teacher signal IS what drives OPD gains | Build T2: VD-aware OPD |
| T1-2 ≈ T1-3 | Vanilla OPD is mostly language-prior distillation | Re-frame paper as negative result OR switch teacher / setting |
| T1-2 better on full-image acc but G_gap flat | Distilling language reasoning, not visual capability | Investigate why; possibly try different student rollout |

**Headline metric**: `Δ = (T1-2 − T1-0) − (T1-3 − T1-0)`, computed per-benchmark and averaged.

## 2. What's NOT built (needs implementation)

### 2.1 Training launcher
- `scripts/train/opd_mmr1_3b_baseline.sh` is a **stub**; rewrite modeled on
  Uni-OPD's `Qwen3_Stu_4B_Tea_A3B_Ins_Math_Code_Uni_OPD.sh` (lines 491-511).
  Pattern: `python ray_launcher.py train.py --flags...`. Take
  `OPD_TEACHER_IMAGE_MODE ∈ {full, blank, both}` as env var so T1-2 and
  T1-3 differ by one knob.
- `configs/baseline/mmr1_3b_vanilla_opd.yaml` exists but Uni-OPD reads
  CLI flags, not YAML. Project-level metadata only; the shell translates.
- `scripts/train/start_teacher_server.sh` references a missing
  `run_sglang_server.sh`; replace with direct
  `python -m sglang.launch_server --model-path ${TEACHER} --port 30000 ...`.
- `teacher_server_list.json` upstream has placeholder; overwrite with
  `{"MMR1-7B-RL": {"path": "...", "servers": ["http://localhost:30000"]}}`.
- `teacher_server_map.json`: set `"default" → "MMR1-7B-RL"`.

### 2.2 BlankTeacher hook (the headline T1-2 vs T1-3 feature)
The intervention point is **`RMSystemManager.build_payload` lines 102-114**:
```python
images: list[Image.Image] | None = sample.multimodal_inputs.get("images", None)
if images:
    payload["image_data"] = [encode_image_for_rollout_engine(img) for img in images]
```
If `OPD_TEACHER_IMAGE_MODE == "blank"`, replace each image with
`Image.new("RGB", img.size, (255,255,255))` before encoding. Student
rollout in `sglang_rollout.py::generate` untouched (always full image).

### 2.3 Per-step diagnostics
A new file `src/mllmopd/training/opd_diagnostics_hook.py` that:
- Wraps Uni-OPD's `post_process_rewards`.
- Per step, for each sample, dumps a JSONL row with: `id`, `position`,
  `teacher_logp_full`, `teacher_logp_blank`, `old_logp_student`,
  `vd = lp_full - lp_blank`, `opd_reward_full = lp_full - student_logp`,
  `opd_reward_blank = lp_blank - student_logp`.
- TensorBoard scalars (aggregates): `mean(vd)`, `frac(vd>0.5)`,
  `frac(loss_mass in high_vd bin)`, `clip_fraction_by_vd_bin`.
- Output path: `${MLLMOPD_RUNS}/<run>/diagnostics/step_{rollout_id}.jsonl.gz`.

### 2.4 Dual teacher scoring (BOTH arms log lp_full AND lp_blank for diagnostics)
Two paths: (A) one teacher server, two `build_payload` calls per sample
per step. ~2x teacher compute, but logp-scoring is prefill-only at
30-50ms/sample on 7B bf16. (B) two teacher servers. Heavier.
**Pick A.** Implemented via env var `OPD_TEACHER_IMAGE_MODE=both` for
both T1-2 and T1-3.

T1-2's *actual reward* uses `lp_full`; T1-3's uses `lp_blank`. The other
quantity is logged for diagnostics only. We need both in both arms so we
can do post-hoc attribution: did T1-3's student converge despite low
`vd = lp_full − lp_blank` (visual signal ignored) or because of it
(anti-visual distillation)? Without dual-logging in T1-3 we can only
compare end-state benchmark accuracy, not the training trajectory.

### 2.5 Training data prep
`scripts/data/prep_opd_train_data.py`:
1. Load `${MMR1_RL_DATA}` via `load_from_disk`.
2. **Multi-level dedup against the WHOLE Level-1 eval subset
   (`data/audit/level1_subset_v0.jsonl`, 1200 prompts)** — NOT just the
   133 `opd_target_ids.json`. T1-eval runs the full 6 × 200 grid, so any
   of those 1200 prompts appearing in training would contaminate eval.
   Dedup layers:
     a. exact id match (cheap sanity check; usually 0 because ids are dataset-local)
     b. normalized question match (lowercased, whitespace-collapsed, punctuation-stripped)
     c. normalized question + normalized choices match (MCQ benchmarks)
     d. image SHA-256 hash match (catches re-asked prompts with reworded text)
     e. fuzzy question match (token Jaccard ≥ 0.85 or MinHash) — log only, don't auto-drop
3. Emit overlap report: counts per layer + per benchmark + sampled examples.
   If layer (b) or (c) overlap exceeds **5%**, **STOP** and switch eval
   to held-out benchmarks (LogicVista / CharXiv / MMMU). Layer (e) > 5%
   is a soft warning, not a stop.
4. Subsample remaining to 2k for T1 v0 (seed-fixed).
5. Emit JSONL with `{prompt, images, label, teacher_model}` matching
   Uni-OPD's `Dataset` expectations (`data.py` lines 297-348).

## 3. Training data choice
**2k subset for T1 v0, 1 epoch**, scale only after numbers are sane.

Budget (matches §6 wall-time below):
- 2k prompts × 8 rollouts/prompt × 1 epoch = 16k rollouts/arm
- Effective batch = 64 → 250 optimizer steps/arm
- ~65s/step × 250 = ~4-5h/arm

If T1-v0 shows a clean Δ signal, T1-v1 escalates to 5k prompts × 1 epoch
or 2k × 3 epochs — that decision is made AFTER T1-v0, not bundled in here.

**Critical**: the whole 1200-prompt Level-1 eval subset must be excluded
from training (see §2.5). Hash by question text + image, since dataset
IDs are dataset-local.

## 5. Eval pipeline for T1 outputs

After each arm produces a checkpoint:
1. **4-mode × 6-benchmark inference grid** via reused `run_audit_pass_sglang.py`.
2. **Aggregate** via `aggregate_audit.py` + `paired_vision_critical.py`.
3. **New analyzer** `src/mllmopd/analysis/t1_compare.py`:
   - Per arm: `G_full[b]`, `G_blank[b]`, `G_gap[b] = G_full[b] - G_blank[b]`,
     `blank_leakage_ratio[b] = G_blank[b] / max(eps, G_full[b])`,
     `opd_target_recovery[b]`.
   - **Headline scalars**: `T1_2_minus_T1_0_full_avg`,
     `T1_3_minus_T1_0_full_avg`, and **Δ = (T1-2 − T1-0) − (T1-3 − T1-0)**.
4. **VD-shift analysis**: re-run `score_completion.py` + `aggregate_vd.py`
   on T1-2/T1-3 outputs. Did the VD distribution shift toward high-VD?
5. **Figures**: extend `reporting/figures.py` with
   - Fig 4: G_full vs G_blank per arm per benchmark
   - Fig 5: Δ scatter across benchmarks for T1-2 minus T1-3
   - Fig 6: VD-NLL-mass histogram, pre vs post training

## 6. Wall time / GPU budget (8×A800-80GB)

Per arm, 2k prompts × 1 epoch:
- **Teacher SGLang server**: 1 GPU (7B bf16, mem_fraction=0.75). Logp-only
  (max_new_tokens=0) is prefill-only ~30 ms/sample.
- **Student rollout**: shared inside Megatron colocated SGLang. 3B bf16,
  ~100 ms/sample. With `n_samples_per_prompt=8`, ~50s per rollout-batch.
- **Trainer**: 7 GPUs Megatron TP=2 DP=3, 3B model, ~5s/optimizer step.
- **Per step**: ~50s rollout + ~5s teacher score + ~10s optimizer ≈ ~65s.
- **Total**: 2000 × 8 / 64 = 250 steps/epoch × 1 epoch ≈ 4-5 hours wall.
- **Both arms**: 8-10 hours.

T1-eval: 12 min × 2 arms = ~25 min.
