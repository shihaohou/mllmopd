## 1. 初心：这个项目到底想回答什么

**核心问题**：vanilla On-Policy Distillation (OPD) 在多模态 LLM (MLLM) 后训练里**什么时候、以什么方式坏掉？**

为什么这个问题值得问：
- 文本侧 OPD 已经被研究透了（SCOPE / TIP / SOD / Prune-OPD 等）
- MLLM 侧的 failure mode **几乎没有系统性 audit**
- 直接跨界搬方法（比如"把 SOD 的 state-drift 思路套到 MLLM"）= 假设了一个**可能根本不存在的问题**
- 所以走 **audit-first**：先观察 vanilla MLLM OPD 在真实任务上具体哪里掉、怎么掉，再决定方法方向

**反面教材**：拿一个 generic 大模型 → distill 到 generic 小模型，比 acc，发论文。这种做法不能告诉我们"OPD 算法本身"什么时候有问题，只能反映"两个模型的差距"。我们刻意避开这种 setup。

**我们的正面 setup**：用同一基座、同一数据生成的 **RL teacher → SFT student** 配对（MMR1-7B-RL → MMR1-3B-SFT）。这样 OPD 注入的"额外信号"和 base model 的 capability gap 是分离的，我们才能干净地测"OPD 到底加了什么、错配了什么"。

## 2. 三条候选假设（都没下注）

| Hypothesis | 一句话陈述 | Validate 信号 | Kill 信号 |
|---|---|---|---|
| **H1 Perception-Reasoning Frontier** | OPD reward 把"看错图"和"推理错"混为一谈 → student 在 perception-hard 题上没学到，但 reasoning-hard 题被过度奖励 | 用 `caption_only_blank` vs `full_image` 分离视觉/推理能力后，OPD 在两类题上 acc 提升不均匀 | OPD 在两类题上 acc 同步上升或下降 |
| **H2 Visual-Signal Misallocation** | OPD loss mass 主要落在**视觉依赖低**的 token 上（也就是"模型已经能猜"的 token）→ 实际学到的不是视觉锚定推理 | 把每个 token 按 `vis_dep = |logp_full − logp_blank|` 分位排序，>50% loss mass 落在最低分位 | loss mass 在 vis_dep 分位上均匀分布 |
| **H3 Post-RL Artifact Transfer** | RL teacher 自带毛病（overthinking / modality shortcut / 长度爆炸），OPD 把这些**和能力一起**继承过来 | OPD 后 student 长度暴涨 + 在简单题上仍 overthink + cross-domain acc 反而下降 | OPD 后 student 长度变化温和 + 能力均匀转移 |

**当前进度**：smoke500 已经在 H3 上看到 baseline 端的可疑信号（teacher 比 student 长 22-62% token 换 8pt acc），但还没跑 OPD 训练所以不能证伪 H3。H1/H2 都需要补能力（captioning pipeline / 强制解码 logprob 抽取）才能 audit。

详见 [`docs/research-plan.md`](research-plan.md) 和 [`docs/experiment-protocol.md`](experiment-protocol.md)。

## 7. 易错点（踩过的坑全列）

按严重度排序，每条都包含**症状 → 原因 → 处理**。

### 7.1 基础设施类

| # | 坑 |
|---|---|
| **1** | **`/root/` 在容器上是 reboot-volatile** —— 损失了一轮 smoke500 数据。`docs/migrate-env.md` Q6 警告过。**Rule**：runs/data 必须在 NFS（`/home/web_server/...`），venv 可以在 nvme（reboot 后 ~10 min 重建可接受） |
| **2** | **`--no-build-isolation` 不会自动装 build deps** —— flash-attn / TE / nvidia-resiliency-ext 都踩过。**Rule**：用 `--no-build-isolation` 之前一律 `uv pip install wheel ninja packaging poetry-core poetry-dynamic-versioning setuptools_scm hatchling maturin cython pybind11` |
| **3** | **PyPI prebuilt wheel 的 torch ABI 不一定匹配你装的 torch** —— flash-attn 2.7.4 PyPI wheel 适配 torch 2.4-2.5，对 torch 2.9.1 直接 `undefined symbol`。**Rule**：训练用的高版本 torch 环境里 flash-attn 必须 `--no-binary flash-attn --no-build-isolation` 从源码编 |
| **4** | **NFS write caching 让 `wc -l` 永远显示 0** —— Python flush 到 OS 后，NFS 客户端还在自己 cache 里。**Rule**：`run_audit_pass.py` 现在每条记录后 `fsync(fileno)`，是必要的 |
| **5** | **UV venv 的 python 是符号链接到 `~/.local/share/uv/python/...`（被 `UV_PYTHON_INSTALL_DIR` 改过路径）** —— tar venv 不一起打包 uv-python 的话 reboot 后符号链接断 → 整个 venv 失效（Q6）。备份时**两个目录都要 tar** |

|---|---|

### 7.3 推理 pipeline 类

| # | 坑 |
|---|---|
| **13** | **`AutoModelForVision2Seq` 对 Qwen2.5-VL 不够** —— 现在 `_build_model` 优先 `Qwen2_5_VLForConditionalGeneration`，并 fallback 到老类 |
| **14** | **flash-attn 装上不等于在用** —— transformers 加载时 `attn_implementation="flash_attention_2"` 才真启用，且必须 ABI 匹配。`_build_model` 用 try/except 让它自动 fallback 到 sdpa |
| **15** | **bf16 7B 解码是 memory-BW bound 不是 compute bound** —— FA2 主要加速 prefill，decode 阶段 FA2 vs sdpa 只快 10-20%。要真加速 decode 必须 **batching**（共享权重读）。当前 A800 7B + decode ~25 ms/token，500 token 输出 = 12-25 s/prompt = 0.04-0.08 prompts/s |
| **16** | **batch=1 时显存利用率 < 30%** —— A800 80GB 浪费 60GB+。`--batch-size 4` 应该 3-4x 加速；batch=8 通常也 OK |
| **17** | **`mllm_corruptions.load` 遇到坏图直接 raise** —— 一张坏图能炸整个 pass。`run_audit_pass` 现在用 try/except 包住，让缺图走 `skip_missing_image` marker row |
| **18** | **`text_only` mode 用 yes/no scorer 时 POPE 全错** —— 模型 refuse 答（"I cannot see image"），parser 抽不出 yes/no → False。这是**模型行为正常**，不是 scorer bug。分析时 text_only POPE cell 要 filter，不能当 grounding 信号用 |
| **19** | **MathVista 的输出经常打满 max_new_tokens=1024** —— 模型在 long CoT 后没生成 EOS。`num_tokens` 字段会显示 1024，注意区分"真长 CoT"和"卡在循环" |

|---|---|
| **26** | **n=20 的 paired 数字几乎是噪声** —— RL 和 SFT 在 20 条样本上有同样的 paired 计数完全是巧合，500 条才开始分化 |
| **27** | **Full−blank acc gap ≠ visual dependency** —— 模型总输出 "(A)" 也是 0 gap。必须看 paired 2x2 (`both_correct / full_only / blank_only / both_wrong`) 和 `image_lift_rate / blank_shortcut_rate`，不能光看 cell-level gap |
| **28** | **scorer=loose_contains 的 cell 不能进图表** —— 它是模糊匹配兜底，可能蒙对。`audit_table` 的 `scorers` 列暴露这个信息，>20% loose 的 cell 标红或排除 |
| **29** | **`acc_per_token` 是粗糙的 efficiency proxy** —— 真效率要在固定难度子集上算，不能跨 benchmark 平均 |
| **30** | **lmms-eval 才是论文级数字** —— 我们的 triage scorer 只用来快速看 pattern，正式 acc 上 Stage E1 跑 lmms-eval |
